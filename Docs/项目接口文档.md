# 项目接口文档

## 1. 文档范围

本文件只记录当前仓库里真实存在、并且仍接入主链路的接口、资源类型和关键类。

以下名字过去出现在旧文档或旧草案里，但当前代码中不应再视为现有接口：

- `TargetingService`
- `TurnFlowController`
- `EnergyState`
- `DeckState`

## 2. 顶层共享接口

### 2.1 `GlobalGameSession`

文件：

- `Scripts/Battle/Shared/GlobalGameSession.cs`

当前职责：

- 保存玩家跨场景共享的最小运行时数据
- 向战斗侧提供玩家名称、生命、移动力和基础攻击参数
- 保存荒川的全局能量状态
- 发出玩家运行时更新信号

当前字段：

- `PlayerDisplayName`
- `PlayerMaxHp`
- `PlayerCurrentHp`
- `PlayerMovePointsPerTurn`
- `PlayerAttackRange`
- `PlayerAttackDamage`
- `ArakawaMaxEnergy`
- `ArakawaCurrentEnergy`
- `PendingMapResumeContext`
- `PendingBattleEncounterId`

当前方法：

```csharp
void SetPlayerCurrentHp(int value)
void SetPlayerMovePointsPerTurn(int value)
void SetArakawaCurrentEnergy(int value)
bool TrySpendArakawaEnergy(int amount)
void RestoreArakawaEnergy(int amount)
void ApplyMovePointDelta(int delta)
void BeginBattle(BattleRequest? request = null)
void SetPendingMapResumeContext(MapResumeContext? resumeContext)
MapResumeContext? PeekPendingMapResumeContext()
MapResumeContext? ConsumePendingMapResumeContext()
void SetPendingBattleEncounterId(string encounterId)
string ConsumePendingBattleEncounterId()
BattleRequest? ConsumePendingBattleRequest()
void CompleteBattle(BattleResult result)
BattleResult? ConsumeLastBattleResult()
Godot.Collections.Dictionary BuildPlayerSnapshot()
void ApplyPlayerSnapshot(Godot.Collections.Dictionary snapshot)
```

当前信号：

```csharp
PlayerRuntimeChanged
ArakawaRuntimeChanged
```

说明：

- 护盾当前不是 `GlobalGameSession` 字段。
- 护盾属于战斗内属性，只存在于战斗对象和运行时状态中。

## 3. 地图原型接口

### 3.1 `IInteractable`

文件：

- `Scripts/IInteractable.cs`

当前职责：

- 统一地图侧交互对象的最小接口
- 为玩家交互检测提供统一入口

当前接口：

```csharp
string GetInteractText(Player player)
bool CanInteract(Player player)
void Interact(Player player)
```

### 3.2 `InteractableTemplate`

文件：

- `Scripts/InteractableTemplate.cs`

当前职责：

- 提供地图交互对象的通用基类
- 处理禁用、冷却和统一交互入口

当前导出字段：

- `DisplayName`
- `PromptText`
- `CooldownSeconds`
- `IsDisabled`

当前接口：

```csharp
virtual string GetInteractText(Player player)
virtual bool CanInteract(Player player)
void Interact(Player player)
protected abstract void OnInteract(Player player)
protected virtual void ApplyCooldown()
```

说明：

- 地图侧推荐的自定义交互对象应继续继承 `InteractableTemplate`，而不是各自重复实现冷却和禁用逻辑。

### 3.3 `Player`

文件：

- `Scripts/Player.cs`

当前职责：

- 地图原型中的可移动玩家角色
- 处理移动、交互范围、视锥筛选和交互优先级

当前导出字段：

- `Speed`
- `Friction`
- `InteractionRange`
- `ViewConeHalfAngleDeg`
- `RequireLineOfSight`
- `InteractionObstacleMask`
- `InteractionStickyWindow`
- `InteractionStickyBonus`
- `ShowInteractionGizmo`
- `GizmoRangeColor`
- `GizmoConeColor`
- `GizmoForwardColor`

当前行为接口：

- 地图移动通过 `_PhysicsProcess(...)`
- 交互输入通过 `_UnhandledInput(...)`

当前额外接口：

```csharp
void ReceiveHeal(int amount)
```

说明：

- 当前 `Player` 仍没有直接发起正式 battle scene 切换。
- 当前 `ReceiveHeal(...)` 已经实现，并会把治疗结果回写到 `GlobalGameSession`。
- `GlobalGameSession` 已能缓存 `BattleRequest` / `BattleResult` / `MapResumeContext`，供地图和战斗之间继续扩展标准边界。

### 3.4 `Npc`

文件：

- `Scripts/Npc.cs`

- 地图侧基础 NPC 交互对象
- 提供最小对话和反馈动画

当前导出字段：

- `NpcName`
- `DialogueText`

### 3.5 `Chest`

文件：

- `Scripts/Chest.cs`

当前职责：

- 地图侧基础宝箱交互对象
- 提供开箱状态和一次性打开逻辑

当前导出字段：

- `ChestName`
- `ItemDescription`

### 3.6 `HealStation`

文件：

- `Scripts/HealStation.cs`

当前职责：

- 地图侧治疗站交互对象
- 按冷却为玩家提供治疗

当前导出字段：

- `HealAmount`
- `CooldownSeconds`

说明：

- 当前实现通过 `player.Call("ReceiveHeal", amount)` 调用玩家恢复逻辑。
- `Player` 当前仓库里没有正式实现 `ReceiveHeal(...)`，所以这条链路仍是原型占位。

### 3.7 `SceneDoor`

文件：

- `Scripts/SceneDoor.cs`

当前职责：

- 地图侧场景切换门
- 负责在交互后切换到目标场景

当前导出字段：

- `NextScene`
- `NextScenePath`
- `BusyText`

当前说明：

- 它当前同时支持两种模式：
  - 普通地图场景切换
  - 发起战斗场景切换
- 进入战斗时会：
  - 写入 `BattleRequest`
  - 写入 `MapResumeContext`
  - 写入待处理 `EncounterId`

### 3.8 `MapSceneController`

文件：

- `Scripts/MapSceneController.cs`

当前职责：

- 在地图场景加载时消费 `MapResumeContext`
- 恢复玩家返回地图时的位置
- 消费最近一次 `BattleResult`

### 3.9 地图原型场景

当前仍存在并可作为地图原型入口/资源的场景包括：

- `Scene/Chest.tscn`

## 10. 地图与战斗边界

当前项目在“地图原型”和“战斗原型”之间的真实边界是：

- 地图侧接口已经存在
- 战斗侧接口已经存在
- 两者已经通过最小 `BattleRequest` / `BattleResult` / `MapResumeContext` 接上玩家快照、失败结果和返回位置
- 当前已经有“地图进入战斗 -> 战斗失败 -> 返回原地图位置”的最小闭环

因此当前不应写成已存在事实的内容包括：

- 完整胜负双向闭环流程控制

如果以后继续扩展，应优先在现有 `BattleRequest` / `BattleResult` 上继续补字段，而不是让地图脚本直接依赖 `BattleSceneController` 内部细节。
