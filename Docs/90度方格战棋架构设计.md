# 90度方格战棋架构设计

## 1. 文档定位

本文件只讨论棋盘战斗子系统，不覆盖地图探索层。地图与战斗的衔接请看：

- `docs/项目总体架构设计.md`
- `docs/项目接口文档.md`

本文件重点解决：

- board 的 object 化建模
- 房间布局初始化
- 单回合内移动与行动结束规则
- 卡牌、能量、荒川强化、索敌系统
- 对障碍物、单位、可通行危险地块的统一管理

## 2. 战斗域的核心目标

你的战斗不是传统“地块 + 单位 + 装饰障碍”模型，而是更接近“棋盘格 + 格上对象集合”的模型。

因此核心目标应为：

- 玩家、敌人、障碍物都作为 `BoardObject` 统一管理。
- 障碍物可以受伤、破损、销毁、转化、生成。
- 障碍物不等于不可通行，有些障碍物可以与单位同格存在。
- 玩家与敌人这类单位不能共享同一格的单位占位槽。
- board 有初始化脚本，接受预设房间布局方案生成战斗场景。
- 攻击、卡牌和荒川能力都基于稳定的格子索引系统与 `BoardObject` 查询系统。

## 3. 现有 Demo 中可复用与必须重构的部分

### 3.1 可复用

- `GameArea` 的坐标换算思路
- `GridCalculator` 的路径和可达域思路
- `RangeCalculator` 的范围计算思路
- `AttackProcessor` / `KnockbackProcessor` 的结算拆分方向
- `BaseSkill` 的数据驱动思路

### 3.2 必须重构

- 现有 `GameGrid.grid_data` 不够支撑多 object 同格和障碍物可受伤。
- 现有 `Unit` 需要拆成逻辑对象和表现对象。
- 现有“移动后进入攻击状态”的流程过于传统，不适合你的一回合多阶段行动。
- 现有技能系统没有覆盖“视线 + 距离 + object 类型筛选”的统一索敌模型。

## 4. 推荐核心模型

## 4.1 Board

`Board` 是战斗中的逻辑棋盘总对象。它至少要包含：

- `BoardTopology`：方格坐标拓扑
- `BoardState`：所有格子逻辑状态
- `BoardObjectRegistry`：所有 object 注册表
- `BoardInitializer`：接收房间布局并生成运行时对象
- `BoardQueryService`：统一查询对象、格子、遮挡、视线、占位

## 4.2 Cell 的推荐结构

每个格子不要只存一个 `unit`，建议至少拆成：

```text
BoardCellState
- cell: Vector2i
- terrain_id: StringName
- height: int
- tags: Array[StringName]
- unit_object_id: StringName?          # 单位占位槽，只允许 0 或 1 个单位型对象
- resident_object_ids: Array[StringName] # 可与单位共存的对象，如电磁地板、陷阱、场域
- blocking_object_ids: Array[StringName] # 可能影响通行或视线的对象
```

这里的重点是：

- `unit_object_id` 只允许一个“单位型 object”。玩家和敌人都属于这一槽位。
- `resident_object_ids` 用于可共存对象，比如电磁地板、火焰场、扫描区、可踩踏陷阱。
- `blocking_object_ids` 用于障碍类 object，但它们不必然完全阻挡通行。

## 4.3 BoardObject

所有场上对象统一抽象为 `BoardObject`。

推荐基础字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `object_id` | `StringName` | 运行时唯一 ID |
| `definition_id` | `StringName` | 对应的定义表 ID |
| `object_type` | `StringName` | `unit` / `obstacle` / `field` / `device` / `pickup` |
| `cell` | `Vector2i` | 当前所在格 |
| `faction` | `StringName` | `player`, `enemy`, `neutral`, `world` |
| `tags` | `Array[StringName]` | 如 `destructible`, `passable`, `conductive`, `cover` |
| `max_hp` | `int` | 可受伤 object 的最大生命 |
| `current_hp` | `int` | 当前生命 |
| `blocks_movement` | `bool` | 是否阻挡移动 |
| `blocks_los` | `bool` | 是否阻挡视线 |
| `stackable_with_unit` | `bool` | 是否可与单位同格 |
| `move_cost_modifier` | `int` | 通过该格时额外消耗 |
| `state_payload` | `Dictionary` | 特殊状态 |

## 4.4 Object 类型建议

### `unit`

- 玩家
- 敌人
- 未来可扩展召唤物

特征：

- 不能与其他 `unit` 共格
- 可以与部分 `obstacle` 或 `field` 共格

### `obstacle`

- 掩体
- 可破坏箱体
- 可导电柱
- 可站上去的电磁地板

特征：

- 可以受伤、破损、销毁
- 有些阻挡移动，有些不阻挡
- 有些阻挡视线，有些不阻挡
- 有些允许单位同格，有些不允许

### `field`

- 电磁场
- 火焰区
- 毒雾区
- 扫描区

特征：

- 通常可与单位同格
- 多数不阻挡移动
- 主要在时间推进或进入格子时触发效果

## 5. 占位与共存规则

推荐把“能否同格”写成明确规则，而不是散落在多个脚本里。

### 5.1 基础规则

- 一个格子最多 1 个 `unit`。
- 一个格子可以有多个 `field`。
- 一个格子可以有多个 `obstacle`，但要限制组合是否合法。
- `stackable_with_unit = true` 的 object 可以与单位同格。
- `stackable_with_unit = false` 的 object 不允许与单位同格。

### 5.2 典型例子

- `电磁地板`：`object_type=obstacle`，`stackable_with_unit=true`，`blocks_movement=false`
- `高掩体箱`：`object_type=obstacle`，`stackable_with_unit=false`，`blocks_movement=true`，`blocks_los=true`
- `地雷`：可设计成 `field` 或 `obstacle`，但应允许单位踩入同格后触发

## 6. Board 初始化与房间布局方案

## 6.1 BoardInitializer

建议新增 [`BoardInitializer.cs`](/D:/Godot/newproject/Card-Chess-Demo/Scripts/Battle/Board/BoardInitializer.cs)，专门负责根据房间布局方案生成 board。

建议接口：

```csharp
void InitializeFromLayout(RoomLayoutDefinition layout, BattleRequest request);
```

## 6.2 RoomLayoutDefinition

房间布局方案不应只是一张 TileMap，而应是逻辑布局定义。

推荐字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `layout_id` | `StringName` | 布局 ID |
| `board_size` | `Vector2i` | 棋盘尺寸 |
| `terrain_paint` | `Array[Dictionary]` | 地形刷子结果 |
| `object_spawns` | `Array[Dictionary]` | 预放置 object |
| `player_spawn_cells` | `Array[Vector2i]` | 玩家出生点候选 |
| `enemy_spawn_cells` | `Array[Vector2i]` | 敌人出生点候选 |
| `tags` | `Array[StringName]` | 如 `tight_corridor`, `sniper_room` |

### `object_spawns` 最少应支持

- `definition_id`
- `cell`
- `rotation`
- `faction`
- `initial_state`

## 6.3 为什么不要把布局初始化写进 Board 本体

因为 `Board` 负责运行时状态，`BoardInitializer` 负责构建。拆开后：

- 更容易测试
- 更容易随机生成房间
- 更容易做固定 Boss 房
- 更容易复用同一棋盘规则层

## 7. 战斗回合结构

你的战斗不是“移动或攻击二选一”，而是“一回合内存在移动阶段 + 主行动阶段”。

推荐把玩家回合状态写成：

```text
TurnActionState
- has_moved: bool
- has_ended_turn: bool
- quick_chain_count: int
- current_turn_index: int
```

## 7.1 玩家基础行动

### 1. 移动

- 由移动力决定本回合能走多远。
- 不结束回合。
- 每回合只能移动一次。
- 移动后仍可普通攻击、防御、冥想或打牌。

### 2. 普通攻击

- 依赖攻击范围和攻击力。
- 必须在物理距离内。
- 必须满足视线条件。
- 结束回合。

### 3. 防御

- 回复护盾值。
- 触发减伤上限限制逻辑。
- 结束回合。

### 4. 冥想

- 弃掉当前所有手牌。
- 抽指定数量卡牌。
- 结束回合。

### 5. 打出卡牌

- 若卡牌有 `迅速`，则不结束回合。
- 否则结束回合。

## 7.2 回合结束判定

回合结束条件应统一由 `TurnFlowController` 判断，而不是分散在卡牌脚本里。

建议逻辑：

- `普通攻击` -> 结束回合
- `防御` -> 结束回合
- `冥想` -> 结束回合
- `打出非迅速卡牌` -> 结束回合
- `移动` -> 不结束回合，但将 `has_moved = true`
- `打出迅速卡牌` -> 不结束回合

## 8. 时间推进系统

## 8.1 核心原则

“时间由玩家行动推进”，而不是由独立实时帧推进。

所以建议新增统一方法：

```gdscript
func advance_time(action_time_cost: int) -> void
```

### 推荐用途

- 推进状态持续时间
- 推进投射物移动
- 推进能量回复计数
- 推进敌人计时行为
- 推进场地效果结算

## 8.2 推荐时间单位

建议定义抽象时间单位 `time_tick`，不要直接把它写死为“回合数”。

例如：

- 移动后的主行动结束时推进 1 tick
- 迅速卡可推进 0 或 1 tick，取决于平衡设计
- 防御推进 1 tick
- 冥想推进 1 tick

## 9. 移动系统设计

## 9.1 移动力模型

玩家属性中应包含：

- `move_points_per_turn`

格子通行消耗由多部分组成：

```text
实际移动消耗 = 地形基础消耗 + object 修正 + 状态修正
```

### 例子

- 普通地面：1
- 厚重金属地：2
- 电磁地板：1 或 2，但附带额外效果
- 被荒川改变后的地块：消耗可动态改变

## 9.2 推荐实现

你提到的做法是对的：

- 使用 A* 求一条路径数组
- 遍历路径逐格累计消耗
- 在达到移动力上限处截断

建议拆成两个方法：

```gdscript
func build_path(start: Vector2i, target: Vector2i) -> Array[Vector2i]
func evaluate_path_cost(path: Array[Vector2i], actor_id: StringName) -> Dictionary
```

返回结果建议包含：

- `total_cost`
- `reachable_cells`
- `truncated_path`
- `entered_objects`

## 10. 普通攻击与索敌系统

## 10.1 统一 TargetingService

普通攻击、卡牌攻击、荒川强化后的攻击都应该走统一索敌服务。

建议至少提供：

```csharp
bool HasLineOfSight(Vector2I fromCell, Vector2I toCell, Godot.Collections.Dictionary parameters);
Godot.Collections.Array<Vector2I> GetCellsByRule(Vector2I origin, StringName ruleId, Godot.Collections.Dictionary context);
Godot.Collections.Array<StringName> GetObjectsInCells(Godot.Collections.Array<Vector2I> cells);
Godot.Collections.Array<StringName> FilterObjects(StringName sourceId, Godot.Collections.Array<StringName> objectIds, StringName ruleId);
```

## 10.2 普通攻击判定

普通攻击必须同时满足：

- 目标在攻击距离内
- 目标在视线内
- 目标符合目标类型筛选
- 目标没有被不允许穿透的 object 完全阻断

## 10.3 视线判定

建议把“视线”和“距离”分开计算。

- 距离：曼哈顿、切比雪夫或专用武器距离
- 视线：沿线采样格子，检查 `blocks_los=true` 的 object

这样后续卡牌能更灵活：

- `冲撞`：无须远程视线，但要求相邻方向
- `狙击`：全图距离，但要求无视障碍或特定穿透规则
- `导弹`：目标点锁定，不一定要求全程视线

## 11. 防御系统

防御不应只是“加护盾”一个字段，建议至少拆为：

- `shield_gain`
- `damage_reduction_cap`
- `duration`

建议由 `DefenseResolver` 或 `DamageResolver` 内的专门方法处理：

```gdscript
func apply_defend(actor_id: StringName, defend_power: int, defend_formula_id: StringName) -> void
func calculate_final_damage(raw_damage: int, target_id: StringName, context: Dictionary) -> int
```

你提到“回复护盾值但限制减伤上限”，这个很适合做成数据驱动公式，而不是硬编码到 UI 或行动脚本里。

## 12. 冥想系统

冥想建议统一走卡牌循环系统，而不是单独手写“删手牌再抽牌”。

推荐流程：

1. 将手牌全部移入弃牌堆
2. 触发弃牌相关效果
3. 抽指定数量卡牌
4. 结束回合

建议接口：

```gdscript
func meditate(draw_count: int) -> void
```

## 13. 卡牌系统

## 13.1 能量系统

能量必须与卡牌系统解耦，建议单独用 `EnergySystem` 或 `BattleResourcePool`。

推荐字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `energy_max` | `int` | 当前能量上限 |
| `energy_current` | `int` | 当前能量 |
| `energy_regen_interval` | `int` | 每多少 tick 回 1 点 |
| `energy_regen_amount` | `int` | 每次回复多少 |
| `energy_regen_counter` | `int` | 当前计数 |

### 解耦要求

因为有卡牌或天赋会改变：

- 能量上限
- 回复间隔
- 单次回复量

所以回复逻辑必须通过统一方法处理：

```gdscript
func on_time_advanced(ticks: int) -> void
func gain_energy(amount: int) -> void
func spend_energy(amount: int) -> bool
func modify_energy_max(delta: int) -> void
```

## 13.2 卡牌循环

卡牌区应至少包含：

- 抽牌堆 `draw_pile`
- 手牌 `hand`
- 弃牌堆 `discard_pile`
- 消耗牌堆 `exhaust_pile`

抽牌规则：

- 抽牌时若抽牌堆为空，立即把弃牌堆洗回抽牌堆
- 若弃牌堆也为空，则停止抽牌

## 13.3 卡牌定义

建议每张卡至少包含：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `card_id` | `StringName` | 卡牌 ID |
| `card_type` | `StringName` | `attack`, `defense`, `skill` |
| `cost_energy` | `int` | 能量消耗 |
| `keywords` | `Array[StringName]` | 如 `quick`, `exhaust` |
| `target_rule_id` | `StringName` | 目标规则 |
| `effect_bundle_id` | `StringName` | 基础效果包 |
| `arakawa_effect_bundle_id` | `StringName` | 荒川强化后效果包 |
| `ends_turn` | `bool` | 默认是否结束回合 |

### 关键词建议

- `quick`：迅速，打出后不结束回合
- `exhaust`：消耗，打出后进入消耗牌堆
- `retain`：可保留
- `ethereal`：若回合结束未打出则消失或进入特定区域

## 13.4 消耗牌堆

你已经明确指出：`消耗` 不能真删除。

这是对的。建议明确为：

- 打出后移入 `exhaust_pile`
- 仍可被某些卡牌、遗物或天赋交互

## 14. 卡牌攻击方式与格子索引管理

你想要的卡牌形态很多，所以必须把“索敌”做成通用系统，而不是写死在每张卡里。

### 14.1 推荐目标模型

卡牌目标规则至少分两层：

1. 几何获取
2. object 过滤

### 14.2 典型攻击模式

#### 冲撞

- 仅上下左右四个方向
- 可能要求相邻格
- 可能推动目标或自身位移

#### 狙击

- 全图或超远距离
- 可定义是否无视障碍物
- 可定义是否必须视线

#### 导弹

- 选择一个格子作为落点
- 对范围内 object 结算
- 可穿越地形
- 可与障碍物强交互

## 14.3 稳定的坐标与索引管理

建议统一使用：

- `cell -> object_ids`
- `object_id -> runtime object`
- `shape query -> object_ids`

不要直接在 UI 或卡牌脚本里遍历场景树找目标。

## 15. 荒川强化

## 15.1 卡牌强化的推荐方式

你已经给出了正确方向：卡牌定义中直接包含“被荒川强化后的效果”。

建议不要在打牌时临时拼接复杂逻辑，而是每张卡保留两套效果：

- 基础效果
- 荒川强化效果

推荐字段：

- `effect_bundle_id`
- `arakawa_effect_bundle_id`
- `arakawa_upgrade_tags`

## 15.2 荒川创造或转换障碍物

由于障碍物也是 `BoardObject`，荒川相关能力可以统一做成：

- 生成新 object
- 修改已有 object 标签
- 转换 object 类型或状态
- 修改地块上的 resident / blocking 集合

这比“直接改 TileMap”更适合你的需求。

## 17. 结论

你这套战斗系统的关键，不是“把单位做出来”，而是先把 `BoardObject + 占位规则 + 房间初始化 + 索敌系统 + 单回合行动规则` 这五块地基打稳。

只要这五块稳了：

- 障碍物受伤与破损
- 可通行障碍物
- 卡牌多样攻击模式
- 荒川改造地形
- 回合内移动后再行动

这些都会自然成立，而不会在后续开发里反复推翻重写。
