# 90度方格战棋架构设计

## 1. 文档依据

本方案基于以下三部分整理：

1. 当前仓库中的 Godot 4.6 等距战棋 Demo 脚本架构。
2. 已成功提取的《游戏开发需求文档.pdf》。
3. 《世界观构建（ai修改版）.pdf》作为补充参考项，但该文件当前因 ObjStm 压缩对象流过重，未在本地环境中稳定提取全文，因此世界观只吸收了需求文档里已明确的内容：太空歌剧、失忆主角、流窜 AI 队友、碎片化叙事。

## 2. 当前 Demo 可复用的结构结论

你现在这套菱形格子 Demo，最有价值的不是等距显示本身，而是它已经把“战棋规则计算”和“节点表现”分出了雏形。可复用点如下：

- `Battle` 作为战斗场景装配点，统一持有移动、范围、攻击、击退、刷怪等工具节点。
- `GameGrid` 维护格子逻辑数据，已经具备“单位占用 + 地形 + 障碍”的基础模型。
- `GameArea` 负责格子坐标与世界坐标互转，这一点对 90 度方格迁移最关键。
- `GridCalculator` 与 `RangeCalculator` 已经把“移动范围”和“技能范围”从 UI 中独立出来。
- `UnitMover`、`AttackProcessor`、`KnockbackProcessor` 分别负责表现执行和结算执行，方向是对的。
- `BaseStateMachine` + `BattleStates` 已经形成“回合流程控制层”。
- `BaseSkill` 走的是资源化配置思路，适合继续演进成卡牌/技能/效果统一框架。

## 3. 当前 Demo 的结构问题

如果直接在现有 Demo 上继续堆功能，后面会很快变得难维护，核心问题有这些：

- `Battle.gd` 同时承担场景装配、战斗上下文、当前单位状态、技能状态、回合重置、死亡清理，职责过重。
- `Unit.gd` 同时承担数值、技能容器、阵营、动画、受伤/死亡逻辑，领域数据和表现耦合过深。
- `GameGrid.grid_data` 把静态地图数据和动态单位占用混在一起，后面做房间生成、地形改变、子弹、机关时会变脆。
- 技能执行直接依赖 `Battle` 上下文，导致技能系统和战斗总控强耦合。
- 当前状态机更像“玩家单位回合流程”，还不足以承载你需求中的“主动即时回合制时间推进”。
- 等距坐标方向、动画方向、网格方向写死在多个脚本里，迁移到 90 度方格后容易出现规则层和表现层一起改的情况。

## 4. 面向 90 度方格战棋的目标设计原则

### 4.1 总体目标

新架构必须满足以下目标：

- 规则层不依赖等距表现，未来可切换为 90 度方格、等距格、六边格。
- 战斗动作统一走“命令 -> 规则结算 -> 表现播放 -> 时间推进”的流水线。
- AI 队友、卡牌、地形改变、投射物、状态效果都能作为独立系统追加，而不是继续塞进 `Battle.gd`。
- 30 到 60 分钟 Demo 能先做出来，但结构上支持后续扩展为完整项目。
- 世界地图、战斗、剧情、成长、构筑、资源系统之间通过数据和事件连接，而不是通过节点互相直连。

### 4.2 推荐分层

建议按五层拆：

1. `App/Flow` 层
负责场景切换、存档、世界地图与战斗切换、局外成长。

2. `Battle Domain` 层
负责战棋规则本体，包括网格、单位、行动、时间轴、卡牌、AI 队友、地形效果。

3. `Content/Data` 层
负责角色模板、卡牌定义、技能定义、敌人模板、房间模板、掉落表、剧情文本键值。

4. `Presentation` 层
负责 TileMap、单位动画、血条、路径预览、受击数字、技能预览、HUD、对话 UI。

5. `Infrastructure` 层
负责事件总线、资源加载、存档序列化、调试工具、日志。

## 5. 推荐模块划分

### 5.1 Battle Flow

- `battle_scene_controller.gd`
  只负责装配战斗场景、初始化依赖、接收上层传入的 Encounter 数据。
- `battle_session.gd`
  持有当前战斗实例的数据引用，不直接写 UI。
- `turn_flow_controller.gd`
  管理阶段切换：玩家输入、指令确认、规则结算、表现播放、时间推进、敌方响应、胜败检查。
- `timeline_system.gd`
  这是你需求里的关键模块。每个行动都有 `time_cost`，行动结束后推进全场时间，并触发持续效果、投射物、敌人计划刷新。

### 5.2 Board / Grid

- `board_topology.gd`
  统一接口，只暴露：相邻格、距离算法、直线采样、范围采样、坐标互转。
- `square_grid_topology.gd`
  90 度方格实现，支持四方向移动、八方向判定、曼哈顿/切比雪夫距离。
- `board_state.gd`
  只存纯逻辑数据：地块、单位占用、机关、危险区域、临时地形改动。
- `board_query_service.gd`
  查询当前位置、可通行性、阻挡、目标合法性。
- `pathfinding_service.gd`
  只做寻路和可达域，不关心动画。
- `targeting_service.gd`
  只做攻击范围、技能范围、卡牌落点、AOE 预览范围。

### 5.3 Unit / Combat

- `actor_model.gd`
  纯逻辑单位数据，字段包括：`actor_id`、阵营、属性、资源、状态、牌组引用、AI 配置引用。
- `actor_view.gd`
  只负责动画、朝向、受击、移动表现。
- `stat_block.gd`
  独立封装 HP、护甲、防御、速度、行动点、AI 能量等数值。
- `damage_resolver.gd`
  统一伤害、治疗、减伤、护盾、击退、死亡判定。
- `status_system.gd`
  中毒、燃烧、易伤、过热、定身、防御等持续状态统一放这里。
- `projectile_system.gd`
  为“主动即时回合制 + 子弹飞行”预留。投射物也属于时间推进对象。

### 5.4 Action / Command

- `battle_command.gd`
  所有行动统一抽象：移动、普攻、防御、打牌、冥想、AI 队友能力。
- `command_validator.gd`
  检查行动是否合法。
- `command_resolver.gd`
  产出 `action_result`，包含位置变化、伤害、资源变化、时间消耗、触发事件。
- `action_result.gd`
  规则层输出的标准结果对象，表现层只消费这个结果。
- `battle_replay_event.gd`
  后续如果你要做回放、战报、战斗日志，这个层会非常省事。

### 5.5 Card / Skill / AI Companion

- `card_definition.gd`
  定义牌费用、目标规则、效果包、强化标签。
- `deck_runtime.gd`
  管理牌库、手牌、弃牌堆、抽牌、洗牌。
- `effect_definition.gd`
  统一技能/卡牌/地形效果的数据描述。
- `effect_executor.gd`
  根据效果数据执行：伤害、位移、召唤、改地形、加状态、抽牌、回复资源。
- `ai_companion_resource_pool.gd`
  管理 AI 队友资源池，和普通法力分开。
- `ai_companion_system.gd`
  封装两类核心能力：地形改变、卡牌强化。后续如果要扩展成剧情互动、战斗中建议系统，也从这里接入。

### 5.6 Encounter / Roguelike

- `encounter_definition.gd`
  定义普通战、精英战、Boss 战、事件战。
- `room_template.gd`
  固定房间模板。
- `room_generator.gd`
  在模板基础上随机生成敌人站位、障碍布局、可破坏物、危险地块。
- `spawn_system.gd`
  只负责根据战斗配置把逻辑 Actor 和表现 Actor 放入场上。

### 5.7 UI / Presentation

- `battle_hud_controller.gd`
  只处理主角五选一行动菜单、手牌、AI 资源、时间轴、提示。
- `grid_overlay_view.gd`
  统一显示可移动区域、攻击范围、技能预览、危险区域。
- `path_preview_view.gd`
  只显示路径。
- `floating_text_view.gd`
  只显示数值飘字。
- `timeline_panel_view.gd`
  显示即将推进的对象、敌方准备动作、投射物刷新顺序。

## 6. 最关键的解耦接口

### 6.1 网格拓扑接口

`BoardTopology` 必须成为规则层唯一认可的“网格定义”。建议最少提供这些方法：

- `world_to_cell(world_pos) -> Vector2i`
- `cell_to_world(cell) -> Vector2`
- `get_neighbors(cell, move_rule) -> Array[Vector2i]`
- `get_distance(a, b, algorithm) -> int`
- `get_cells_in_range(center, shape, params) -> Array[Vector2i]`
- `get_direction(from, to) -> Vector2i`

这样你从菱形格迁移到 90 度方格时，只需要替换 `Topology` 和对应的 View，不需要重写战斗规则。

### 6.2 统一行动接口

每次玩家或敌人的操作都不要直接改场景节点，而是先生成命令：

- `MoveCommand`
- `AttackCommand`
- `DefendCommand`
- `PlayCardCommand`
- `MeditateCommand`
- `UseAICompanionCommand`

然后统一交给 `command_resolver`。这样后面加敌方 AI、录像回放、撤销调试、战斗日志会容易很多。

### 6.3 表现和规则分离

规则层只关心：

- 单位从 A 格到 B 格
- 目标受到多少伤害
- 哪个地块被改成危险地形
- 哪张牌被强化
- 时间推进了多少

表现层只关心：

- 播放移动动画
- 播放攻击动画
- 更新 HUD
- 播放地形变化特效
- 更新路径预览和时间轴

## 7. 适配你需求的核心战斗循环

建议战斗主循环改为：

1. 玩家选择一个行动。
2. 生成 `BattleCommand`。
3. `CommandValidator` 校验目标、资源、时间消耗是否合法。
4. `CommandResolver` 结算行动，输出 `ActionResult`。
5. `PresentationQueue` 播放这一步动作的视觉表现。
6. `TimelineSystem` 根据 `time_cost` 推进时间。
7. 刷新敌方意图、投射物、持续状态、房间机关。
8. 检查胜败、掉落、事件触发。

这套循环直接对应你的“主动即时回合制”。它比传统我方回合/敌方回合的状态机更适合你当前题材。

## 9. 从当前 Demo 到新架构的迁移映射

| 当前脚本 | 保留价值 | 迁移目标 |
| --- | --- | --- |
| `scenes/battle/battle.gd` | 装配入口 | 拆成 `battle_scene_controller.gd` + `battle_session.gd` |
| `scenes/battle/game_area/game_area.gd` | 坐标转换 | 变成 `square_grid_topology.gd` 的 View 适配层 |
| `scenes/battle/game_grid/game_grid.gd` | 格子数据模型 | 拆成 `board_state.gd` + `board_query_service.gd` |
| `scripts/grid_calculator.gd` | 移动范围与路径 | 改为 `pathfinding_service.gd` |
| `scripts/range_calculator.gd` | 技能/攻击范围 | 改为 `targeting_service.gd` |
| `scripts/unit_mover.gd` | 移动表现 | 改为 `actor_motion_view.gd` 或 `presentation_queue` 子模块 |
| `scripts/attack_processor.gd` | 基础伤害公式 | 改为 `damage_resolver.gd` |
| `scripts/knockback_processor.gd` | 位移结算经验 | 改为 `effect_executor.gd` 中的位移效果模块 |
| `data/skills/base_skill.gd` | 数据驱动技能 | 改为 `card_definition.gd` + `effect_definition.gd` |
| `scenes/states/*` | 流程状态切换 | 压缩为 `turn_flow_controller.gd`，让更多逻辑转向命令流水线 |

## 10. 与世界观和叙事的接口预留

为了让你后续把太空歌剧世界观顺利接进来，建议数据层额外预留：

- `encounter_theme_id`：废弃飞船、科研遗迹、货运站、外星遗址。
- `dialogue_trigger_id`：战前、战后、探索物件、Boss 首次遭遇。
- `ai_companion_comment_pool`：AI 队友在战斗内外的动态吐槽与提示。
- `memory_fragment_id`：碎片叙事收集项。
- `terrain_style_id`：保证不同战斗房间能统一切换视觉主题，而不改规则逻辑。

## 11. Demo 优先级建议

### P0 必须先做

- 90 度方格移动与寻路
- 主角五选一行动
- 时间推进系统
- 基础敌人 AI
- 基础卡牌使用
- AI 队友两种核心能力
- 房间模板 + 伪随机敌人与障碍生成
- 战斗胜负与基础奖励

### P1 建议加入

- 状态效果
- 投射物系统
- 简单局外成长
- 构筑界面
- 固定 Boss 房
- 碎片化文本叙事入口

### P2 可后续扩展

- 复杂地形联动
- 多 AI 队友人格
- 高级敌方行为树
- 战斗日志回放
- 高级事件房和路线分支

## 12. 结论

这套方案的核心不是“把菱形格改成正方格”，而是把你的项目从“能跑的战斗原型”升级成“规则层稳定、表现层可换、系统能持续加”的结构。

如果你按这个拆法继续做，后面无论加卡牌、AI 队友、地形改造、子弹时间推进，还是从 Demo 扩成完整项目，都不需要推翻底层。
