# 90度方格战棋需求表

## 1. Demo 目标摘要

- 类型：JRPG + 90 度方格战棋 + 卡牌构筑。
- 主题：我的 AI 队友。
- 世界观基调：太空歌剧、失忆主角、流窜 AI、碎片化叙事。
- 平台：PC。
- 引擎：Godot 4.6.x。
- Demo 时长：30 到 60 分钟。
- 核心目标：完整证明战斗机制可玩，世界观有吸引力，系统可以继续扩展。

| 普通敌人模板 | 3 | 近战、远程、功能型各一种 |
| Boss 模板 | 1 | 固定地图战使用 |

| 房间模板 | 5 到 8 个 | 普通房随机抽取 |
| Boss 房模板 | 1 到 2 个 | 固定或半固定 |
| 地块类型 | 4 到 6 种 | 普通、阻挡、危险、可改造、特殊机关 |

## 4. 战斗动作需求表

| 动作 | 作用 | 资源消耗 | 时间消耗建议 | 备注 |
| --- | --- | --- | --- | --- |
| 移动 | 调整站位 | 无 | 1.0 | 可受地形影响 |
| 普通攻击 | 基础输出 | 无 | 1.0 | 可触发基础反击或状态 |
| 防御 | 降低后续受伤 | 无 | 1.0 | 可持续到下次行动前 |
| 打出卡牌 | 主要策略来源 | 卡牌费用 | 1.0 到 1.5 | 允许强效果但要控制资源 |
| 冥想 | 弃牌补牌 | 无 | 1.0 | 原地罚站，形成节奏决策 |
| AI 地形改造 | 改战场 | AI 资源 | 1.0 到 2.0 | 应有强烈战术价值 |
| AI 卡牌强化 | 放大卡牌上限 | AI 资源 | 0.5 到 1.0 | 可设计为即时辅助行为 |

## 5. 非功能需求

| 类别 | 要求 |
| --- | --- |
| 可扩展性 | 新增卡牌、敌人、地形、AI 能力时，尽量只新增资源文件和单个模块脚本 |
| 可维护性 | 规则层不直接依赖具体场景节点，尽量通过数据对象和结果对象通信 |
| 可调试性 | 每步行动输出结构化日志，便于排查时间推进和资源结算问题 |
| 可平衡性 | 卡牌、敌人、AI 队友能力尽量数据驱动，避免写死在流程脚本里 |
| 表现稳定性 | 所有移动、攻击、地形改造、投射物结算都要可视化且不阻塞主循环 |
| 兼容 Demo 周期 | 所有 P0 需求都应以最小可运行版本优先，不为“完整大系统”过早投入过多成本 |

## 6. 推荐开发阶段

### 第一阶段：战斗骨架

- 完成 90 度方格地图、寻路、单位站位。
- 建立 `BattleCommand -> ActionResult -> Presentation` 基础流水线。
- 打通移动、攻击、防御、胜败结算。

### 第二阶段：你的核心卖点

- 接入时间推进系统。
- 接入牌库/手牌/弃牌/费用。
- 接入 AI 队友资源、地形改造、卡牌强化。
- 做出至少 3 类普通敌人。

### 第三阶段：Demo 完整度

- 接入房间随机生成。
- 接入构筑和少量全局成长。
- 做 1 个 Boss 房。
- 接入剧情文本和战后奖励。

### 第四阶段：打磨

- 补危险区域提示、敌方意图提示、时间轴 UI。
- 调整卡牌与 AI 资源数值。
- 修复卡死、穿模、结算顺序问题。

## 7. 当前最值得优先落实的工程任务

1. 先把 `BoardTopology` 和 `BoardState` 从当前 `GameArea` / `GameGrid` 中拆出来。
2. 再把 `Move / Attack / Defend / PlayCard / Meditate / UseAICompanion` 统一抽成命令。
3. 然后实现 `TimelineSystem`，因为这是你项目和普通战棋最大的差异点。
4. 最后再接卡牌、AI 队友、房间随机化，不要一开始就把所有系统一起写死在战斗场景里。
