# 战斗对外交互接口方案

## 1. 目标

在地图、战斗、成长、构筑、奖励、存档都尚未完全收敛之前，先固定 battle 对外的接口边界。

核心原则：

- 地图只负责“发起战斗”和“消费战斗结果”
- 战斗只负责“消费战斗请求”和“产出战斗结果”
- 成长 / 构筑 / 背包 / 奖励 / 存档都不直接耦合战斗场景内部实现
- 所有跨域交互都通过 DTO、全局会话和服务接口完成

这份文档的重点不是 UI，而是契约、责任边界和隐藏实现的隔离。

## 2. 总体结论

### 2.1 是否需要全局角色状态脚本

需要。

但不建议写成一个什么都做的“超级脚本”。更合理的做法是：

- 用一个顶层 `GlobalGameSession` 作为总入口 / 外观层
- 由它持有若干清晰分工的运行时状态对象

建议结构：

```text
GlobalGameSession
  PartyRuntimeState
  ProgressionRuntimeState
  DeckBuildState
  InventoryRuntimeState
  PendingBattleState
  SaveRuntimeState
```

也就是说：

- `GlobalGameSession` 是对外唯一入口
- 真实数据按领域拆到子状态对象中
- 地图和战斗只依赖 `GlobalGameSession` 暴露的标准接口

### 2.2 当前最重要的接口目标

先稳定 6 条链路：

1. 地图遭遇 -> 战斗请求
2. 全局状态 -> 战斗初始化
3. 构筑状态 -> 战斗卡组初始化
4. 战斗结果 -> 奖励 / 成长增量
5. 战斗失败 -> 返回存档点
6. 存档系统 -> 保存 / 加载全局状态

## 3. 顶层模块划分

建议以后按下面的责任来划：

### 3.1 地图层

负责：

- 地图交互
- 敌人 / 门 / 事件触发
- 获取 `EncounterId`
- 调用进入战斗接口
- 返回地图后消费战斗结果

不负责：

- 拼 battle 内部对象
- 计算奖励
- 改成长细节
- 改卡组运行时细节

### 3.2 战斗层

负责：

- 消费 `BattleRequest`
- 选 encounter
- 解析房间池
- 初始化房间、棋盘、战斗对象
- 初始化战斗卡组运行时
- 结算战斗过程
- 产出 `BattleResult`

不负责：

- 地图节点恢复
- 背包入库
- 专精点分配
- 永久成长计算
- 存档落盘

### 3.3 成长层

负责：

- 主角成长
- 队友荒川成长
- 天赋 / 专精 / 解锁
- 影响战斗外和战斗内的长期成长状态

对战斗提供：

- 战前成长快照
- 战后成长增量应用接口

### 3.4 构筑层

负责：

- 战斗外卡牌构筑
- 牌库组成
- 初始遗物 / 天赋导致的 deck modifier
- 特殊开局规则

对战斗提供：

- 一个可初始化 battle deck runtime 的标准快照

### 3.5 奖励层

负责：

- 根据遭遇、房间、事件和战斗结果生成奖励
- 产出道具、经验、专精点、解锁、剧情标记

不应该放到 `BattleSceneController` 内。

### 3.6 存档层

负责：

- JSON 序列化和反序列化
- 自动存档和手动存档槽位策略
- 存档版本迁移

不直接参与战斗逻辑，只提供保存和恢复能力。

## 4. 推荐的全局状态结构

## 4.1 `GlobalGameSession`

建议作为 battle 这边唯一直接依赖的全局入口。

建议职责：

- 提供当前主角 / 队友运行时状态
- 提供成长、构筑、背包的只读快照入口
- 缓存待处理的 `BattleRequest`
- 缓存最近一次 `BattleResult`
- 缓存地图恢复上下文
- 暴露统一的结果应用方法

不建议把所有字段全平铺在它自己身上。

当前代码状态：

- `GlobalGameSession` 已开始按这个方向收敛
- 当前内部已挂接：
  - `PartyRuntimeState`
  - `ProgressionRuntimeState`
  - `DeckBuildState`
  - `InventoryRuntimeState`
  - `SaveRuntimeState`
- 现阶段仍保留一层旧字段镜像，用于兼容现有逻辑和 inspector 配置

## 4.2 `PartyRuntimeState`

建议管理：

- 主角当前 HP / MaxHP
- 主角战斗基础参数
- 荒川当前状态
- 队友战斗参数
- 是否已入队

建议最少包含：

```csharp
PlayerRuntimeSnapshot Player
CompanionRuntimeSnapshot Arakawa
Godot.Collections.Array<CompanionRuntimeSnapshot> ReserveCompanions
```

当前代码状态：

- `PlayerRuntimeState`
- `CompanionRuntimeState`
- `PartyRuntimeState`

这三层骨架已经落地。

## 4.3 `ProgressionRuntimeState`

建议管理：

- 主角等级 / 经验
- 专精点
- 已解锁天赋
- 已解锁构筑选项
- 队友成长节点

当前代码状态：

- `ProgressionRuntimeState` 骨架已落地
- 当前最小承载字段包括：
  - 等级
  - 经验
  - 专精点
  - 天赋 id
  - 荒川解锁 id

## 4.4 `DeckBuildState`

建议管理：

- 当前战斗外构筑牌组
- 构筑版本号
- 解锁卡牌池引用
- 初始遗物 / 天赋对 deck 的常驻修正

注意：

- 这不是 battle runtime deck
- 这是战斗开始前的“构筑输入”

当前代码状态：

- `DeckBuildState` 骨架已落地
- 当前最小字段为：
  - `CardIds`
  - `RelicIds`

## 4.5 `InventoryRuntimeState`

建议管理：

- 可堆叠道具
- 装备 / 遗物
- 关键物品
- 奖励待领取队列

当前代码状态：

- `InventoryRuntimeState` 骨架已落地
- 当前最小实现为 `ItemCounts`

## 4.6 `SaveRuntimeState`

建议管理：

- 当前手动存档槽元信息
- 自动存档元信息
- 当前复活 / 失败回退用的最近存档点

当前代码状态：

- `SaveRuntimeState` 骨架已落地
- 当前最小字段为：
  - `LastCheckpointSaveId`
  - `LastManualSaveId`

## 5. 地图 -> 战斗的入口方案

你现在的思路是对的，建议正式固定为：

```text
地图交互脚本
  -> 取得 EncounterId
  -> 由 EncounterId 找 EncounterProfile
  -> 由 EncounterProfile 找 RoomPool
  -> 构建 BattleRequest
  -> 写入 GlobalGameSession
  -> 切换到 Battle 场景
```

## 5.1 推荐显式边界

地图只传这些：

- `EncounterId`
- `MapResumeContext`
- `BattleRequest`

不要让地图把房间实例直接传给战斗。

建议：

- 地图只传 `EncounterId`
- 战斗侧根据 `EncounterId` 查 `BattleEncounterLibrary`
- `BattleEncounterProfile` 再解析：
  - `EnemyTypeIds`
  - `PrimaryEnemyDefinitionId`
  - `PreferredRoomPoolId`

这样 battle 仍然掌握房间抽取和 encounter 解释权。

## 5.2 为什么房间不应该由地图直接实例化

因为一旦地图直接传 battle room prefab：

- 地图会依赖战斗房间资源细节
- 地图需要知道 battle 内部资源组织方式
- 后续房间池、权重、限制条件变更时，地图也要改

这会破坏分层。

## 6. 战斗初始化方案

战斗开局建议只依赖一个完整的 `BattleRequest`。

当前代码状态：

- `BattleRequest` 已提供 `TryValidate(out string failureReason)`
- `GlobalGameSession.BeginBattle(...)` 已会拒绝非法 request

## 6.1 `BattleRequest` 推荐扩展

建议后续扩展为：

```csharp
string RequestId
string EncounterId
int RandomSeed
Godot.Collections.Dictionary PlayerSnapshot
Godot.Collections.Dictionary CompanionSnapshot
Godot.Collections.Dictionary ProgressionSnapshot
Godot.Collections.Dictionary DeckBuildSnapshot
Godot.Collections.Dictionary DeckRuntimeInitOverrides
Godot.Collections.Dictionary RuntimeModifiers
```

各字段建议：

- `RequestId`
  - 用于 battle result 回写时校验配对
- `EncounterId`
  - 用于 encounter / reward / room pool 解析
- `RandomSeed`
  - 用于确定性复现
- `PlayerSnapshot`
  - 主角即时战斗参数
- `CompanionSnapshot`
  - 荒川或其他队友即时参数
- `ProgressionSnapshot`
  - 战前长期成长状态
- `DeckBuildSnapshot`
  - 战斗外构筑
- `DeckRuntimeInitOverrides`
  - 特殊开局牌堆初始化
- `RuntimeModifiers`
  - 地图事件、天气、特殊房间效果等临时修正

## 6.2 卡组初始化为什么要单独有 `DeckRuntimeInitOverrides`

因为你提到这几个需求都不是普通构筑本身能表达的：

- 起手加入指定卡
- 弃牌堆预置卡牌
- 消耗堆预置卡牌
- 手牌张数修改
- 开局抽牌规则变化
- 天赋导致的初始状态变化

所以推荐把 deck 初始化拆成两层：

1. `DeckBuildSnapshot`
   - 表示“战斗外我带了什么牌”
2. `DeckRuntimeInitOverrides`
   - 表示“这一场战斗开局时怎么摆到各个牌堆里”

## 6.3 `BattleDeckState` 未来应具备的初始化能力

当前 `BattleDeckState` 还主要是从固定 `startingDeck` 开始。

后续建议支持：

```csharp
BattleDeckRuntimeInit
{
	BuildCards
	StartingHandCards
	StartingDrawPileCards
	StartingDiscardPileCards
	StartingExhaustPileCards
	HandSizeOverride
	InitialEnergy
	DrawCountOnBattleStart
	ShuffleSeed
}
```

## 7. 战斗结果方案

## 7.1 `BattleResult` 推荐扩展

建议最终扩展为：

```csharp
string RequestId
string EncounterId
BattleOutcome Outcome
Godot.Collections.Dictionary PlayerSnapshot
Godot.Collections.Dictionary CompanionSnapshot
Godot.Collections.Dictionary ProgressionDelta
Godot.Collections.Dictionary InventoryDelta
Godot.Collections.Array RewardEntries
string ClearedEncounterId
Godot.Collections.Dictionary RuntimeFlags
```

当前代码状态：

- `BattleResult` 已提供 `TryValidate(out string failureReason)`
- `GlobalGameSession.CompleteBattle(...)` 已会拒绝非法 result

推荐 `BattleOutcome`：

- `Victory`
- `Defeat`
- `Retreat`
- `Abort`

其中 `Retreat` 的约定为：

- 玩家主动选择逃跑动作成功脱离战斗
- 战斗结果仍需要回写主角和荒川当前状态
- 默认不标记 `ClearedEncounterId`
- 默认不按胜利流程结算奖励

## 7.2 胜利时

战斗只负责：

- 产出战后快照
- 产出奖励意图 / 增量
- 标记清理的 encounter

战斗不要自己：

- 直接发经验
- 直接加专精点
- 直接往背包塞东西
- 直接存档

这些都交给外部系统消费 `BattleResult` 后处理。

## 7.3 失败时

你提的“失败返回上一个存档点”是合理的，但建议失败结果里不要直接塞“切换场景逻辑”。

推荐：

- `BattleResult.Outcome = Defeat`
- 地图 / 全局系统收到后查询“最近合法回退点”
- 再决定：
  - 自动存档点回退
  - 手动存档点回退
  - 是否给玩家选择

也就是说：

- battle 输出失败事实
- save / session 系统决定怎么回滚

## 7.4 逃跑时

逃跑建议视为和 `Atk`、`Def`、`Meditate` 同级的正式动作，而不是地图层强行切场景。

推荐规则：

1. 玩家在 `PlayerAction` 中点击 `Retreat`
2. 当前单位原地停留并结束本回合
3. 在随后的敌方回合结束前，如果玩家 `HP` 没有下降，则逃跑成功
4. 护盾损失不算失败条件，只有 `HP` 下降才算逃跑失败
5. 成功时 battle 产出 `BattleOutcome.Retreat`
6. 外部系统消费 `BattleResult` 后返回地图，并回写当前主角 / 荒川状态

这样设计的原因是：

- 逃跑仍然是战斗规则的一部分
- battle 不需要知道地图里怎么处理逃跑后事件
- 地图 / 成长 / 存档系统只需要识别 `Outcome = Retreat`

## 8. 奖励与成长方案

## 8.1 奖励不建议在 battle 内硬编码

你提到：

- 奖励可能和房间布局预制件有关
- 奖励会通往背包系统
- 奖励也可能影响成长系统

这意味着奖励至少要由两个 id 共同决定：

- `EncounterId`
- `RoomLayoutId` 或 `RoomTemplateId`

可选再加：

- 战斗评级
- 特殊事件 flag
- 是否无伤

## 8.2 推荐奖励解析方式

推荐增加一个 battle 外部服务：

```csharp
RewardBundle ResolveBattleRewards(
	string encounterId,
	string roomLayoutId,
	BattleOutcome outcome,
	Godot.Collections.Dictionary runtimeFlags)
```

这样：

- `BattleSceneController` 只负责产出基础 result context
- `RewardResolver` 决定最终奖励包

## 8.3 奖励包建议内容

```csharp
RewardBundle
{
	int Experience
	int MasteryPoints
	Godot.Collections.Array<ItemRewardEntry> Items
	Godot.Collections.Array<CardRewardEntry> CardRewards
	Godot.Collections.Dictionary ProgressionDelta
}
```

## 8.4 成长层接口建议

成长系统建议只暴露：

```csharp
ProgressionSnapshot BuildProgressionSnapshot()
void ApplyProgressionDelta(ProgressionDelta delta)
```

其中 `ProgressionDelta` 可以包含：

- `exp_delta`
- `mastery_point_delta`
- `unlock_flags`
- `talent_unlocks`
- `companion_growth_delta`

## 9. 构筑系统方案

卡牌构筑是战斗外强相关系统，必须单列接口。

## 9.1 构筑系统对战斗的唯一输出

建议是：

```csharp
DeckBuildSnapshot BuildBattleDeckSnapshot()
```

里面只描述：

- 主牌组列表
- 可带的起始 relic / module / passive 影响
- 构筑版本

## 9.2 构筑系统不应该做的事

不应该：

- 直接控制 battle 中的 draw / discard / exhaust 运行时状态
- 直接访问 `BattleDeckState`

这些是 battle runtime 层负责的。

## 10. 存档系统方案

## 10.1 推荐拆成独立存档服务

建议增加：

```csharp
SaveGameService
```

职责：

- 将 `GlobalGameSession` 序列化为 JSON
- 从 JSON 恢复 `GlobalGameSession`
- 管理手动槽位和自动槽位
- 做版本迁移

## 10.2 推荐存档槽策略

### 手动存档

- 多槽位
- 玩家主动在固定地点保存

### 自动存档

- 单独 1 个槽位
- 不占用手动槽
- 在关键地点或关键流程触发

## 10.3 战斗与存档的边界

战斗系统不负责直接写 JSON。

更合理的流程：

```text
战斗结束
  -> 生成 BattleResult
  -> GlobalGameSession 应用结果
  -> SaveGameService 根据策略决定是否自动保存
```

## 10.4 失败回退与存档的关系

建议显式区分：

- `LastSafeCheckpoint`
- `LastManualSave`
- `LastAutoSave`

失败时由全局系统决定回退策略，不由战斗场景决定。

## 13. 可能出现的问题与风险

## 13.1 双系统并存风险

现在仓库里已经出现过：

- 旧地图链
- 新 battle 链

同名概念但不同实现并存的情况。

问题：

- 很容易出现脚本引用串线
- 场景跑的是 A 链，autoload 却是 B 链

建议：

- battle 这边后续只围绕 `GlobalGameSession + BattleRequest + BattleResult`
- 不再让旧系统类型渗入 battle 主链

## 13.2 DTO 过度膨胀风险

如果把所有字段都平铺到 `BattleRequest` / `BattleResult`：

- 维护成本很高
- 序列化不稳定
- 字段责任不清

建议：

- 保持顶层 DTO 稳定
- 把复杂内容拆成子快照 / 子增量

## 13.3 奖励归属不清风险

如果 battle 既负责胜负、又负责奖励、又负责成长：

- `BattleSceneController` 会越来越重
- 地图 / 成长系统后续不好替换

建议：

- battle 只产出结果上下文
- reward / progression 在外部结算

## 13.4 存档回退时机风险

如果失败时由战斗场景直接决定“回哪个存档”：

- battle 会知道太多 save 细节
- 自动 / 手动槽逻辑会污染 battle

建议：

- battle 只返回 `Defeat`
- session / save 层决定回退

## 13.5 构筑和 runtime deck 混淆风险

这是后续非常容易踩坑的一点。

必须区分：

- 构筑牌组：战斗外长期状态
- runtime 牌堆：战斗内临时状态

如果混成一个结构：

- 战斗外改卡组会污染战斗内逻辑
- 战斗中临时改动又会错误回写到长期构筑

## 13.6 奖励表与房间池耦合风险

你提到奖励可能和房间布局预制件一起关联，这是可以的，但不要把奖励直接写死在房间场景脚本里。

建议：

- 房间只提供 `LayoutId` / `RoomTag`
- 奖励系统通过 `EncounterId + LayoutId` 再做解析

## 13.7 逃跑结果归类错误风险

如果把逃跑成功错误地当成 `Victory`：

- 会污染奖励发放
- 会错误清理 encounter
- 会让地图侧无法区分“打赢”与“脱离”

建议：

- 始终使用 `BattleOutcome.Retreat`
- 地图 / 奖励 / 成长按 `Outcome` 分流处理

## 13.8 接口协作原则

后续多人协作时，建议明确遵守：

- 对接出问题时，优先调整自己这一侧的实现代码
- 不要第一时间修改 `BattleRequest` / `BattleResult` / `GlobalGameSession` 等公共接口
- 只有在双方确认接口结构本身存在问题时，才允许修改接口定义

这样做的原因是：

- 能减少“两边同时改接口”带来的未知问题
- 能保持主链接口稳定
- 能让地图、战斗、成长、存档各自并行推进

## 14. 建议的最终责任边界

### 地图侧看见的接口

```csharp
BattleRequest BuildBattleRequest(string encounterId)
bool TryEnterBattle(BattleRequest request, MapResumeContext resumeContext, out string failureReason)
BattleResult? ConsumeLastBattleResult()
```

### 战斗侧看见的接口

```csharp
BattleRequest? ConsumePendingBattleRequest()
void CompleteBattle(BattleResult result)
PartyRuntimeSnapshot BuildPartySnapshot()
DeckBuildSnapshot BuildDeckBuildSnapshot()
ProgressionSnapshot BuildProgressionSnapshot()
```

### 成长 / 奖励 / 存档侧看见的接口

```csharp
ProgressionSnapshot BuildProgressionSnapshot()
void ApplyProgressionDelta(ProgressionDelta delta)
void ApplyInventoryDelta(InventoryDelta delta)
RewardBundle ResolveBattleRewards(...)
void SaveToSlot(SaveSlotKind slotKind)
```

## 15. 结论

你的整体判断是对的：

- 需要统一管理主角和荒川状态的全局脚本
- 需要把构筑、奖励、成长、存档都提前纳入接口设计
- 需要把地图遭遇 -> 房间池 -> 战斗 -> 奖励 -> 成长 / 背包 / 存档这条链路拆成标准边界

真正关键的不是现在把所有系统做完，而是：

- battle 输入结构先固定
- battle 输出结构先固定
- 全局状态、构筑、奖励、成长、存档都通过明确接口接入

只要这层契约先定稳，后面你和朋友分别推进实现，最终对接是可控的。
