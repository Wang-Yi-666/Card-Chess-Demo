# 项目总体架构设计

## 1. 文档范围

本文件描述当前仓库里已经落地的项目结构，而不是理想中的完整成品设计。

当前工程实际是两套并行原型：

- 地图交互原型
- 独立战斗原型

两者目前共存在同一个 Godot 工程里，但还没有做成完整的“地图进入战斗再返回地图”的闭环。

## 2. 当前入口

- `project.godot` 当前主场景：`Scene/Battle/Battle.tscn`
- 全局 Autoload：`GlobalGameSession`

这意味着现在启动项目时，默认进入的是战斗测试场景，而不是地图探索场景。

## 3. 当前总分层

### 3.1 全局共享层

`Scripts/Battle/Shared/GlobalGameSession.cs`

当前只负责保存玩家运行时的少量共享数据：

- `PlayerDisplayName`
- `PlayerMaxHp`
- `PlayerCurrentHp`
- `PlayerMovePointsPerTurn`

它已经能发出 `PlayerRuntimeChanged` 信号，但还不是完整的存档、牌组、地图进度容器。

### 3.2 地图交互原型层

当前地图侧主要是早期原型脚本与场景：

- `Scripts/Player.cs`
- `Scripts/Npc.cs`
- `Scripts/Chest.cs`
- `Scripts/HealStation.cs`
- `Scripts/SceneDoor.cs`
- `Scripts/InteractableTemplate.cs`
- `Scripts/IInteractable.cs`

对应场景主要有：

- `Scene/Mainlevel.tscn`
- `Scene/NpcScene.tscn`
- `Scene/Chest.tscn`
- `Scene/Healing.tscn`
- `Scene/SceneDoor.tscn`
- `Scene/BaseInteractable.tscn`

这些内容目前仍然是独立地图交互原型，没有与战斗域建立标准数据接口。

### 3.3 战斗原型层

当前战斗域集中在 `Scripts/Battle`、`Scene/Battle`、`Resources/Battle`。

已存在的实际模块为：

- `Board`
- `Data`
- `Rooms`
- `Actors`
- `State`
- `Presentation`
- `Visual`
- `UI`
- `Turn`
- `Shared`

## 4. 当前战斗启动链路

当前实际运行流程如下：

1. `BattleSceneController` 在 `_Ready()` 中创建 `BoardState`、`BoardObjectRegistry`、`BoardQueryService`、`TurnActionState`。
2. 控制器从 `BattleRoomPoolDefinition` 或导出的房间场景中选择一个 `BattleRoomTemplate`。
3. `BattleRoomTemplate.BuildLayoutDefinition()` 把标记层转成 `RoomLayoutDefinition`。
4. `BoardInitializer.InitializeFromLayout(layout)` 初始化棋盘与对象注册表。
5. `BattleObjectStateManager` 根据 registry、prefab library 和 `GlobalGameSession` 生成运行时状态。
6. `BattlePieceViewManager` 把对象状态实例化成 prefab 视图。
7. `BattleBoardOverlay` 负责可达区域、高亮和预览路径。
8. `BattleHudController` 负责底部状态条、调试按钮和结束回合入口。

## 6. 当前已实现的战斗能力

- `BoardObject` 统一管理玩家、敌人、障碍物
- 单位槽与非单位对象集合的占位模型
- 房间模板经 marker layer 初始化棋盘
- 玩家移动力、可达范围和简化路径预览
- 每回合最多移动一次
- 显式结束回合 / 进入下一回合
- prefab 表现层与基础动画状态
- HUD 状态显示与调试按钮

## 7. 当前未实现的系统

以下内容在文档里曾被当成既定架构，但在当前代码中还不存在：

- `BattleRequest`
- `BattleResult`
- `MapResumeContext`
- 地图进入战斗并返回地图的流程控制器
- `TurnFlowController`
- `TargetingService`
- 普通攻击、防御、冥想
- 卡牌系统
- 能量系统
- 时间推进系统
- 存档 / 读档

## 8. 当前一致性结论

如果以后继续写文档，应以“当前项目是独立战斗原型 + 独立地图交互原型”作为前提，而不能再把完整总循环写成已实现事实。

更准确的说法是：

- 地图原型已经存在
- 战斗原型已经存在
- 两者尚未通过标准请求结果结构正式接通
