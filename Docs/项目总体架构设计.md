# 项目总体架构设计

## 1. 目标

本文件描述的是整个项目的总运行逻辑，而不是仅限于棋盘战斗。

你的游戏应被视为一个由两个核心玩法域组成的项目：

1. 地图探索域
2. 棋盘战斗域

玩家的完整循环是：

`地图探索 -> 遇敌/事件 -> 进入棋盘战斗 -> 战斗结算 -> 返回地图 -> 继续探索`

因此，项目架构必须从一开始就把“地图层”和“战斗层”拆开，并通过稳定的数据接口交接，而不是让地图直接操控战斗节点，或者让战斗直接回写地图场景对象。

## 2. 整体设计原则

- 地图系统和战斗系统各自独立开发、独立测试。
- 两者只通过“战斗入口数据”和“战斗结算结果”交互。
- 地图层负责探索、遇敌、补给、拾取、世界事件。
- 战斗层负责棋盘战斗、卡牌、荒川能力、时间推进、敌方行为。
- 局外持久数据由统一的 `GameSession` 或等价运行时对象管理。
- 你朋友负责地图逻辑时，不需要了解战斗内部实现；你负责战斗逻辑时，也不应依赖地图内部节点结构。

## 3. 总体分层

建议整项目按六层划分：

1. `App / Flow`
2. `World Map Domain`
3. `Battle Domain`
4. `Meta Progression / Persistent State`
5. `Content / Data`
6. `Presentation / UI`

### 3.1 App / Flow

负责全局流程切换：

- 启动游戏
- 进入地图
- 触发战斗
- 加载战斗场景
- 接收战斗结果
- 返回地图
- 存档/读档

### 3.2 World Map Domain

负责地图探索阶段的规则：

- 地图移动
- 遇怪判定
- 交互点
- 补给点
- 道具拾取
- 地图事件
- 地图状态变化

### 3.3 Battle Domain

负责棋盘战斗阶段的规则：

- 90 度方格战斗
- 回合内行动
- 时间推进
- 卡牌与荒川能力
- 敌人 AI
- 战斗结算

### 3.4 Meta Progression / Persistent State

负责跨地图和跨战斗保留的数据：

- 玩家属性成长
- 当前牌组
- 荒川手提箱能量库存
- 已获得道具
- 已触发剧情
- 通缉/扫描风险等全局标记

### 3.5 Content / Data

负责静态数据：

- 地图模板
- 交互点定义
- 道具定义
- 敌人定义
- 卡牌定义
- 房间模板
- 战斗遭遇模板
- 对话文本

### 3.6 Presentation / UI

负责所有视觉层表现：

- 地图 UI
- 战斗 HUD
- 对话 UI
- 飘字
- 时间轴
- 荒川投影

## 4. 核心循环

推荐总流程如下：

1. 玩家进入地图场景。
2. 地图系统驱动移动、交互与事件。
3. 当玩家接触到敌对触发器或进入战斗事件点时，地图层生成 `BattleRequest`。
4. `GameFlowController` 保存当前地图上下文并切换到战斗场景。
5. 战斗系统根据 `BattleRequest` 初始化棋盘、敌人、环境和可用资源。
6. 战斗结束后生成 `BattleResult`。
7. `GameFlowController` 将 `BattleResult` 回传地图层与全局会话。
8. 地图层根据结果更新敌人消失、道具掉落、剧情推进、补给状态等。
9. 玩家继续探索。

## 5. 最关键的地图层与战斗层接口

### 5.1 BattleRequest

地图进入战斗时，只传入一个标准请求对象，建议包含：

- `encounter_id`
- `map_node_id`
- `enemy_group_id`
- `battlefield_theme_id`
- `room_rule_tags`
- `player_runtime_snapshot`
- `deck_snapshot`
- `arakawa_energy_snapshot`
- `fuel_snapshot`
- `global_flags`
- `reward_table_id`

### 5.2 BattleResult

战斗结束后，只返回一个标准结果对象，建议包含：

- `victory`
- `defeat_reason`
- `hp_delta`
- `status_changes`
- `deck_changes`
- `arakawa_energy_delta`
- `fuel_delta`
- `loot_items`
- `new_flags`
- `removed_map_enemy_id`
- `triggered_story_ids`

### 5.3 MapResumeContext

返回地图时，需要恢复地图上下文，建议包含：

- 玩家回到地图的位置
- 当前地图节点状态
- 已清除的敌人点
- 已使用的补给点
- 已拾取道具
- 当前地图阶段性剧情状态

## 6. 模块边界与职责分工

### 6.1 你负责的战斗域

建议由你负责：

- 战斗场景控制器
- 棋盘逻辑
- 卡牌系统
- 荒川战斗能力
- 时间推进
- 敌方战斗 AI
- 战斗结算和 `BattleResult`

### 6.2 你朋友负责的地图域

建议由你朋友负责：

- 地图探索逻辑
- 地图移动与镜头
- 遇怪触发
- 地图交互点
- 手提箱能量补充站
- 地图道具拾取
- 地图事件节点
- 地图返回点恢复

### 6.3 双方的协作边界

双方共享但不混写的部分：

- `GameSession`
- `BattleRequest`
- `BattleResult`
- 角色持久数据结构
- 道具定义数据
- 荒川资源定义
- 全局剧情 Flag

## 8. 总运行时状态设计

建议有一个统一的 `GameSession` 保存跨场景状态。

### 8.1 GameSession 至少应保存

- 当前地图 ID
- 当前地图进度
- 玩家 HP / 属性
- 当前牌组
- 当前手提箱燃料库存
- 当前荒川能量上限或相关成长值
- 已获得道具
- 已解锁剧情 Flag
- 当前追踪/扫描风险

### 8.2 为什么不能让地图和战斗各存一份

如果地图和战斗各自维护一套玩家状态，后面很容易出现：

- 地图上拾取的道具没有传入战斗
- 战斗里消耗的能量没有回写地图
- 战斗受伤结果无法正确同步
- 剧情 Flag 在两个场景内冲突

所以必须以 `GameSession` 为唯一事实源。

## 9. 地图层需要特别考虑的系统接口

虽然地图架构由你朋友负责，但为了保证战斗层可接入，以下接口必须提前约定。

### 9.1 遇怪接口

地图层不直接加载某个具体战斗场景，而是：

- 识别敌人触发器
- 选择 `encounter_id`
- 组装 `BattleRequest`
- 调用 `GameFlowController.start_battle(request)`

### 9.2 补给站接口

手提箱能量补充站不应直接写某个战斗脚本变量，而应写入：

- `GameSession.fuel_inventory`
- 或 `GameSession.arakawa_resource_state`

这样战斗初始化时自然能读取到。

### 9.3 道具拾取接口

地图上的道具拾取建议统一走：

- `item_definition`
- `inventory_system.add_item(item_id)`
- 触发必要的 `flag` 或数值变化

如果是战斗前即时生效型道具，再由 `BattleRequest` 快照带入战斗。

## 10. 战斗层需要特别考虑的回写接口

战斗结束后，战斗层不应该直接去操作地图对象，只应该返回结果。

地图层根据 `BattleResult` 处理：

- 清除敌人点
- 播放地图掉落
- 启用新交互点
- 解锁剧情文本
- 开启补给站
- 更新追踪风险

## 11. 世界观如何分布到地图与战斗

### 11.1 更适合地图层承载的内容

- 星际联盟铁幕监管氛围
- 星港扫描与黑户压力
- 手提箱能量补给站
- 物品描述与记忆碎片拾取
- 与荒川的非战斗对话
- 地图事件与分支

### 11.2 更适合战斗层承载的内容

- 搭船客战斗单位
- AI 猎手的压制机制
- 荒川地形改造
- 电弧武器强化
- 时间推进与战场危险

### 11.3 两边都要共享的内容

- 主角非法身份
- 荒川资源
- 手提箱燃料
- 阵营关系
- 剧情推进 Flag

## 12. 总体开发顺序建议

### 第一阶段：打通总循环

- 建立 `GameFlowController`
- 建立 `GameSession`
- 定义 `BattleRequest` 与 `BattleResult`
- 做出最简地图场景到战斗场景再返回地图的完整闭环

### 第二阶段：地图和战斗各自最小可玩

- 地图层完成移动、遇怪、拾取、补给点
- 战斗层完成移动、攻击、防御、胜负结算

### 第三阶段：主题系统接入

- 接入卡牌
- 接入荒川资源
- 接入手提箱地形改造
- 接入地图扫描事件与黑户风险

### 第四阶段：内容扩充与打磨

- 房间随机
- 敌人扩展
- Boss 战
- 剧情文本
- 构筑与成长

## 13. 结论

这份总架构的重点是把项目看成“地图探索游戏 + 战棋战斗游戏”的双域系统，而不是只看棋盘内一场战斗。

只要你们双方都围绕 `GameSession`、`BattleRequest`、`BattleResult` 这三层接口开发，就可以做到：

- 地图和战斗并行开发
- 模块责任清晰
- 后续扩展不会互相拉扯
- 你朋友可以安全地写地图层
- 你可以专注战斗层而不被地图实现细节绑死
