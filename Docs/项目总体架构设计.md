# 项目总体架构设计

## 1. 目标

本文件描述整个项目的总运行逻辑，而不是仅限于棋盘战斗。

完整循环是：

`地图探索 -> 遇敌/事件 -> 进入棋盘战斗 -> 战斗结算 -> 返回地图 -> 继续探索`

新版本的重点补充是：战斗层中的 board 采用 `BoardObject` 统一对象模型，因此地图层传给战斗层的遭遇请求，必须能够兼容房间布局定义、对象生成规则和初始化参数。

## 2. 整体设计原则

- 地图系统和战斗系统独立开发。
- 两者只通过标准数据接口交互。
- `GameSession` 是跨场景唯一事实源。
- 战斗层内部使用 `BoardObject` 模型，不暴露给地图层场景节点。
- 地图层只关心“触发何种遭遇”，不关心战斗里对象如何实例化。

## 3. 总体分层

1. `App / Flow`
2. `World Map Domain`
3. `Battle Domain`
4. `Meta Progression / Persistent State`
5. `Content / Data`
6. `Presentation / UI`

## 4. 地图与战斗的交接原则

### 4.1 地图层负责

- 地图移动
- 遇怪触发
- 交互点
- 补给站
- 道具拾取
- 世界事件
- 组装 `BattleRequest`

### 4.2 战斗层负责

- 根据房间布局定义初始化 board
- 在 board 上生成玩家、敌人、障碍物等 `BoardObject`
- 执行移动、攻击、防御、冥想、打牌、时间推进
- 输出 `BattleResult`

## 5. 战斗层在总体架构中的新关键点

由于你已经明确“玩家、敌人、障碍物都定义为 board 上的 objects”，所以总架构层需要承认以下事实：

- 战斗初始化不是简单地把单位摆上 TileMap，而是构建一份运行时 object 图。
- 房间布局方案属于内容层的重要输入，而不是纯表现资源。
- 可破坏障碍、可通行障碍、场地效果 object 都可能参与战斗结算与掉落。

## 6. 推荐总流程

1. 玩家在地图中移动。
2. 地图层检测到敌人触发器或事件点。
3. 地图层组装 `BattleRequest`，其中包含遭遇、房间主题、敌人组、玩家快照等信息。
4. `GameFlowController` 切换到战斗场景。
5. 战斗层读取 `BattleRequest`，选择 `RoomLayoutDefinition`。
6. `BoardInitializer` 生成 board、cell 状态和 object 注册表。
7. 玩家执行战斗，战斗层输出 `BattleResult`。
8. `GameFlowController` 回写 `GameSession`，生成 `MapResumeContext`。
9. 地图层恢复状态并继续探索。

## 7. BattleRequest 在总架构中的最低要求

除原有字段外，建议强调：

- `layout_id` 或 `room_layout_pool_id`
- `enemy_group_id`
- `battlefield_theme_id`
- `random_seed`
- `player_snapshot`
- `deck_snapshot`
- `arakawa_snapshot`
- `suitcase_snapshot`

## 8. BattleResult 在总架构中的最低要求

建议强调：

- `victory`
- `hp_delta`
- `inventory_delta`
- `arakawa_energy_delta`
- `fuel_delta`
- `new_flags`
- `cleared_encounter_id`
- `triggered_story_ids`

## 9. GameSession 在总架构中的新增要求

因为障碍物、房间和卡牌会更复杂，`GameSession` 至少还应稳定保存：

- 当前牌组状态
- 手提箱燃料库存
- 荒川可用资源相关成长值
- 与房间触发相关的全局 Flag
- 地图中已清除遭遇和已消耗交互点

## 10. 协作边界

### 10.1 你朋友不需要关心的内容

- `BoardObject` 如何注册
- 障碍物如何受伤和破损
- 视线和索敌如何计算
- 卡牌如何调用 targeting

### 10.2 你不需要关心的内容

- 地图角色控制
- 补给站场景表现细节
- 地图上的道具拾取交互 UI
- 地图事件节点的镜头与演出逻辑

### 10.3 需要共同对齐的内容

- `BattleRequest`
- `BattleResult`
- `GameSession`
- `MapResumeContext`
- 资源命名规范
- 剧情 Flag 命名规范

## 12. 总体里程碑更新

### M1 总循环闭环

- 地图可触发一场战斗
- 战斗结束可返回地图
- 状态能正确同步

### M2 战斗地基闭环

- board 可根据房间布局初始化
- BoardObject 可统一管理玩家、敌人、障碍物
- 玩家可完成移动、普通攻击、防御、冥想

### M3 核心卖点闭环

- 卡牌系统接入
- 能量系统接入
- 荒川强化接入
- 可破坏障碍与对象同格规则接入

### M4 内容扩展

- 更复杂房间
- 更多 object 类型
- Boss 战
- 扫描事件与剧情推进

## 13. 结论

总架构层现在最重要的新增约束是：战斗层不再是“单位和地形分离”的老式结构，而是“由房间布局驱动的 board object 运行时系统”。

只要地图层继续只负责发请求，战斗层只负责消费请求并回传结果，你们两人依然可以保持解耦开发。
